# Docs Assets, Starter-Docs State, and Session History

> Filename: `2026-04-20-docs-assets-state-and-history.md`. See `docs/.references/design-contract.md` for naming and structural rules.

## Purpose

Define the intended documentation architecture for moving starter-docs operational state out of `docs/.starter-docs/`, introducing a template-owned `docs/.assets/` namespace, and migrating agent session history out of `docs/guides/agent/`.

This is a design-only step. It does not implement CLI changes, move existing files, update templates, or create the new template directories. It establishes the reviewable source of truth for a later plan and work backlog.

## Context

The current starter-docs CLI writes its install manifest to `docs/.starter-docs/manifest.json` and stages update conflicts under `docs/.starter-docs/conflicts/`. That means running the CLI creates a documentation subdirectory that is not part of the template itself. For a tool whose job is to install and synchronize a coherent docs system, CLI-managed state should live inside a documented, template-owned namespace.

The current docs contract also treats agent session records as guides under `docs/guides/agent/`. That has been useful, but it blurs the distinction between stable user/developer guidance and operational history generated by agents while maintaining the project.

The replacement model should solve the immediate manifest location issue without over-designing future capabilities. It should also leave a clear path for related operational documentation assets such as agent history, memories, preferences, and other non-guide records.

## Design Lineage

This design updates the direction set in [Docs Contract v2 - Planning Proposals](2026-04-15-docs-contract-v2-planning.md), specifically the prior decision to store agent session guides under `docs/guides/agent/`.

The new decision keeps the W/R/P lineage concept, but moves generated session records into an assets namespace and formalizes a single combined coordinate field for wave, revision, phase, stage, and task positioning.

## Decision

Introduce `docs/.assets/` as the template-owned home for operational documentation assets and CLI-managed docs state.

### Directory Model

Future template shape:

```text
docs/
  .assets/
    AGENTS.md
    CLAUDE.md
    starter-docs/
      manifest.json
      conflicts/
        <run-id>/
          ...
    history/
      YYYY-MM-DD-<slug>.md
```

The `docs/.assets/` directory is an operational documentation namespace. It is part of the docs system, but its contents are not ordinary guides, designs, plans, PRDs, or work items.

`docs/.assets/AGENTS.md` and `docs/.assets/CLAUDE.md` should act as routers. They should explain that this directory contains operational documentation assets, direct agents to the relevant subdirectory, and keep each subdirectory's instructions minimal and purpose-specific.

`docs/.assets/starter-docs/` is reserved for starter-docs CLI state. The canonical manifest path becomes:

```text
docs/.assets/starter-docs/manifest.json
```

Future conflict staging should move to:

```text
docs/.assets/starter-docs/conflicts/<run-id>/...
```

`docs/.assets/history/` becomes the canonical home for agent session records, replacing `docs/guides/agent/`. The `docs/guides/` tree should remain focused on stable user-facing and developer-facing guides.

Potential future namespaces such as `docs/.assets/memories/` and `docs/.assets/preferences/` are explicitly deferred. They should not be created until there is a concrete contract for their purpose, routing, schema, and lifecycle.

### History Records

Future agent history records should use flexible YAML frontmatter. The schema is measurement-friendly, not rigid. A record should include fields when the session knows them and omit fields when it does not.

Recommended fields when known:

```yaml
---
client: "Codex Desktop"
model: "gpt-5.4"
date: "2026-04-20"
coordinate: "W3 R0 P1"
---
```

Useful optional dimensions:

```yaml
provider: "OpenAI"
repo: "starter-docs"
branch: "main"
status: "completed"
summary: "Moved starter-docs state and history design into reviewable form."
```

The `coordinate` field replaces separate `wave`, `revision`, `phase`, `stage`, and `task` fields. Do not split those values into separate frontmatter properties unless a later contract explicitly changes this decision.

### Coordinate Convention

Coordinates should use one combined abbreviation that captures as much positioning as is known for the session:

```yaml
coordinate: "W3 R0 P1"
```

The base format is:

```text
W{wave} R{revision} P{phase}
```

When deeper granularity is known, append stage and task suffixes:

```yaml
coordinate: "W3 R0 P1 S2 T4"
```

The hierarchy is:

- A wave may contain multiple revisions.
- A revision may contain multiple phases.
- A phase may contain multiple stages.
- A stage may contain multiple tasks.

The combined abbreviation should capture the known coordinate for the session. Unknown deeper levels should be omitted rather than filled with placeholders.

### Alpha Install Treatment

Existing alpha installs do not need legacy migration support from `docs/.starter-docs/`. The follow-on implementation can treat the new path as canonical without preserving a compatibility reader for old alpha manifests, unless a later review explicitly changes that scope.

## Alternatives Considered

### Move the Manifest to `docs/.references/`

This would avoid creating an untemplated `.starter-docs` directory, but it would mix mutable CLI state with stable reference material. `docs/.references/` should remain a place for reusable guidance and contracts, not install metadata or conflict staging.

### Keep `docs/.starter-docs/` and Add Better Documentation

This would minimize code churn, but it would leave the core concern unresolved: running the CLI would still create a state directory that is not part of the template's documented directory model.

### Move Only the Manifest and Leave History in `docs/guides/agent/`

This would solve the immediate CLI state concern, but it would miss the related structural issue that generated agent history is not the same kind of artifact as a guide. Moving both into `docs/.assets/` gives the docs system one coherent home for operational documentation assets while keeping future namespaces optional.

### Design a Full Assets Platform Now

This would create more structure than the project currently needs. The design intentionally limits first-scope decisions to starter-docs state and session history, while naming `memories/` and `preferences/` only as deferred possibilities.

## Consequences

The follow-on implementation will need to update starter-docs manifest and conflict paths in the CLI, tests, smoke scripts, and user/developer documentation that currently mention `docs/.starter-docs/`.

The docs template will need to add `docs/.assets/` with router instructions and subdirectory-specific instructions for `starter-docs/` and `history/`.

History contracts and templates will need to change from `docs/guides/agent/` to `docs/.assets/history/`. The history template should include YAML frontmatter with the flexible schema described here, including the combined `coordinate` field.

Root and guide router instructions will need to stop describing agent session records as guides. They should route history work to `docs/.assets/history/` and keep `docs/guides/` focused on stable user and developer guides.

Existing historical records under `docs/guides/agent/` should be moved by the follow-on implementation only after the reviewed plan decides whether to move all current records in one phase or split current-record migration from template/CLI changes.
